## 一、完整版（项目经历段落，约 200 字）

**WeChat Solo — 多 Agent 人设驱动的单机微信模拟器**（2026.07 · 独立项目）

- **S（背景）**：为验证 AI 产品助理的「产品定义 + AI 落地」能力，独立设计并实现一个高保真微信 Demo，核心差异化是用 Agent 人设替代真实好友。
- **T（任务）**：完成从 PRD 到代码的完整闭环：信息架构设计、5 个 Agent 人设与规则引擎、聊天/通讯录/朋友圈/个人中心四大模块、数据持久化与主题系统。
- **A（行动）**：用 React 19 + TypeScript + Zustand + Dexie（IndexedDB）搭建纯前端架构；设计规则驱动的 Agent 回复引擎，模拟「正在输入」、延迟、已读等真实社交细节；深色模式采用 CSS 变量主题，页面零改动全局换肤；全程以 Claude Code 子代理驱动开发，遵循「设计文档 → 任务拆解 → TDD → 独立审查」流程。
- **R（结果）**：5 个 Sprint 全部交付，35 个测试文件 119 个用例全通过；项目可离线运行、可现场演示，代码与文档已开源至 Gitee。

## 五、高频追问预演

**Q：Agent 回复为什么不接真实大模型？**
A：作品集演示需要稳定可控——真实 LLM 的延迟和不确定性会干扰面试演示。规则驱动反而更能展示人设设计和 Prompt 工程思维，且每个角色风格可预测，面试官能快速感知差异。架构上引擎与规则库解耦，后续接 LLM 只需替换规则匹配层。

**Q：深色模式怎么做的？**
A：没用 Tailwind 的 `dark:` 变体——那要在每个元素上补类名，改动面大易遗漏。改为 CSS 变量主题：tailwind 色板引用 `var(--wechat-*)`，`[data-theme="dark"]` 统一覆盖，页面代码零改动全局换肤，设置持久化到 IndexedDB。

**Q：最大的技术难点是什么？**
A：一是种子数据的确定性——初期用随机 ID 和时间戳导致测试不稳定，改为计数器 ID + 固定基准时间后测试可复现；二是子代理协作的质量控制——遇到过子代理误删共享样式、报告与实际提交不符的问题，后来建立了「核实真哈希 + 审查删除量 + 全分支回归」的验收机制。

**Q：如果继续做，下一步是什么？**
A：接入真实 LLM 替换规则引擎（架构已预留替换点）、群聊与联系人标签（已在优先级规划 P2）、图片消息与查看器。
