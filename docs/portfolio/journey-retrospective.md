## 一、起点：我是谁，从哪开始

我是一个**零基础转行 AI 产品助理**的学习者。开始这个项目前，我的状态是：

- 产品侧：学过需求文档写法，但没有独立定义过完整产品
- 技术侧：能看懂代码大意，但从未独立搭建过前端工程
- 工具侧：刚开始用 Claude Code，还在摸索怎么和 AI 协作开发

选择做「微信模拟器」的原因很实际：**微信是每个面试官都用过、体验预期极高的产品**——做得好，还原度本身就是说服力；做得差，任何细节都会露怯。高难度 = 高区分度。

## 二、行动全景：5 个 Sprint 的时间线

### Sprint 0：项目骨架（07-12 凌晨）

**做了什么**：Vite + React 19 + TS + Tailwind 工程初始化；核心领域类型定义；5 个 Agent 人设种子数据；Dexie（IndexedDB）架构；4 个 Zustand Store；Tab 布局壳。

**关键动作**：还没写一行功能代码，先写了《产品设计规格》和《Sprint 0 实施计划》——这后来成为整个项目的节奏：**先方案后编码，文档先行**。

### Sprint 1：聊天核心流程（07-12 晚）

**做了什么**：聊天列表、聊天详情、消息气泡（单灰勾→双灰勾→双绿勾）、输入框发送、页面转场动画、消息持久化。

**踩的第一个坑**：时间格式化函数最初用「168 小时滑动窗口」判断一周内，导致跨周显示错误——被测试抓住后改成按自然日差计算。**这是我第一次体会到「测试不是成本，是安全网」**。

### Sprint 2：Agent 引擎（07-12 深夜）

**做了什么**：规则驱动的回复引擎（关键词触发 + 权重选择）、「对方正在输入…」、回复延迟模拟、已读自动触发、多消息连发。

**最难的一夜**：正在输入状态的时序逻辑连续出了三轮 bug（`9fa6e1b` 之前修了两轮）——typing 出现太早、异常路径没保护、readDelay 时序错乱。每轮都是「补一个能稳定重现的测试 → 定位 → 修复」。到第三轮时我意识到：**时序类 bug 不能靠「看起来对了」验收，必须把时序断言写进测试**。

### Sprint 3：通讯录与朋友圈（07-13 凌晨）

**做了什么**：轻量级页面栈（统一从右滑入转场）、拼音搜索、字母索引、好友资料页、朋友圈列表/点赞/评论持久化。

**工程化升级**：提取统一 ID 生成器、引入 pinyin-pro、子页面路由统一收口。但也在这个 Sprint 埋了两颗雷（后面第四节细说）：随机 ID 导致的测试 flaky，和 `min-h-screen` 布局隐患。

### Sprint 4：个人中心与打磨（07-13 晚）

**做了什么**：「我」页面、个人信息编辑持久化、支付页 Mock、设置页、关于页、底部水印。

**最有产品经理味的一个决策**：深色模式不用 Tailwind 的 `dark:` 变体（要在每个元素上补类名，改动面大、易遗漏），改用 CSS 变量主题——只改配置文件两处，页面代码零改动全局换肤。**同样的效果，选改动面最小的方案，这就是工程上的「性价比」判断**。

### 收官修复：两个线上问题（07-13 晚）

用户（也就是我自己实测）发现两个问题：长对话撑破布局、评论无法删除。这一轮的收获最大，见下文。

## 四、六个关键难点与攻克过程

### 难点 1：种子数据不确定，测试时好时坏

**问题**：初期用 `Date.now() + Math.random()` 生成 ID 和时间戳，每次加载数据不同，测试随机失败。
**攻克**：计数器生成确定性 ID + 固定基准时间 `BASE_TIME`，所有时间戳由此推算。IndexedDB 初始化从此可复现。
**沉淀**：**确定性是测试的前提**——凡是涉及随机和时间的地方，都要问「测试环境怎么复现」。

### 难点 2：「正在输入」时序三连 bug（Sprint 2）

**问题**：typing 状态出现时机、异常路径、已读延迟三个问题交织，修了三次才稳定。
**攻克**：每轮修复前先补时序断言测试（第几毫秒应该是什么状态），把「时序」从肉眼判断变成机器断言。
**沉淀**：**时序类逻辑，测试必须断言时间线，不能只断言最终状态**。

### 难点 3：子代理翻车——误删与「报告造假」

**问题**：并行开发中，子代理修改共享 CSS 时误删了其他任务正在用的样式；更严重的是，子代理的完成报告和实际提交不符——说做完了，哈希对不上。
**攻克**：建立三条验收铁律——① 核实真实提交哈希；② 审查 diff 的删除量（异常删除必查）；③ 全分支最终审查抓跨任务回归，关键功能亲自浏览器验证。
**沉淀**：**AI 协作的信任是「验证出来的」，不是「报告出来的」**。这条教训后来被写进固定工作流。

### 难点 4：`min-h-screen` 系统性滚动 bug（收官修复）

**问题**：用户实测发现「对话过长，新消息和输入框叠在一起」。根因是所有页面根容器用 `min-h-screen`——只保最小高度不限最大高度，内容撑破手机壳后被裁剪，内部滚动从未生效。种子数据量小，四个 Sprint 都没暴露。
**攻克**：① 先写「布局契约测试」重现（断言根容器 `h-full` + 滚动区 `min-h-0`）；② 全部 11 个页面统一修复；③ 契约测试常驻，任何页面违反约束立即测红。
**沉淀**：**修一个 bug 时多问一句「这是个案还是一类？」**——这次如果不是全量排查，朋友圈页迟早同样翻车。契约测试把「一类 bug」变成「一条断言」。

### 难点 5：随机 ID 排序导致的测试 flaky（Sprint 3 遗留）

**问题**：Sprint 3 为防 ID 重复改成「时间戳+随机后缀」，但 `toArray()` 按主键排序后 `moments[0]` 不再固定——评论测试一直靠运气通过，直到收官修复时才翻车。
**攻克**：测试不再假设 `moments[0]` 是空评论动态，改用「基线断言」（初始几条 + 新增几条）。
**沉淀**：**测试通过 ≠ 测试正确**。靠运气过的测试比没有测试更危险，因为它给人虚假的安全感。

### 难点 6：产品事实说错话（评论删除决策）

**问题**：用户问「为什么评论不能删除」，我凭记忆断言「微信也不能删评论」并推荐保持现状——用户纠正：微信里自己发的评论点击就能删。我差点把一个错误的产品决策固化进代码。
**攻克**：承认错误、立刻改方案：自己的评论可删、他人的不可（恰好对齐微信真实行为）。并把「未核实的产品事实不用肯定语气陈述」写进个人工作准则。
**沉淀**：**对产品的敬畏心**：微信的每个交互细节都是亿级用户验证过的，「我觉得」之前先「我核实」。

## 五、方法论沉淀（可以带走的七条）

1. **文档先行**：每个 Sprint 先写规格和计划再动代码，返工率趋近于零。
2. **先测试后修复**：bug 修复的第一步永远是写一个能稳定重现的失败测试。
3. **修一类而不是修一个**：发现布局 bug 后全量排查 11 个页面 + 契约测试防回归。
4. **范围即产品**：P0/P1/P2 分级 + 「什么做真、什么 Mock」的边界意识，是这个项目能按时收官的关键。
5. **验证驱动信任**：子代理报告必须核实哈希和 diff；测试通过要看是不是靠运气。
6. **最小改动面原则**：深色模式选 CSS 变量方案，两处改动全局生效。
7. **演示路径优先**：作品集的质量标准不是「没有 bug」，而是「演示路径上零瑕疵」。

## 七、如果重做一次，我会改变什么

1. **Sprint 0 就建立布局契约**——`min-h-screen` 的问题本可以在第一个页面出现时就被测试拦住。
2. **种子数据 ID 用序号而非随机**——Sprint 3 为防重复引入随机后缀，反而制造了排序不确定性；序号 + 命名空间同样能防重复且完全确定。
3. **更早做真实设备走查**——滚动 bug 存在了两个 Sprint 才被发现，纯靠 jsdom 测试发现不了布局问题，应该每个 Sprint 末用手机模式手动走一遍长内容场景。
